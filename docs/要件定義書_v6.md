## 🎯 プロジェクト概要

**個人用・非公開の家計簿アプリ。**

### 解決する課題

既存の家計簿アプリはクレジットカード支払いの反映に数日かかる。  
これを**即日（即時）反映**することで、リアルタイムの金銭管理を実現する。

### コアの目的

**食費の予算オーバーを防ぐこと。**

- クレカのみで生活しているが、従来アプリだとカード反映が数日後になり予算の進捗がわからない
- 食費がメインだが、生活費全般も記録・管理したい
- このアプリは「予算オーバーを防ぐブレーキ」かつ「お金の流れを把握する記録」として使う

### 開発目的

- アプリの完成
- **Railsの学習**（Active Storage / Action Mailer / Active Job / Sidekiq などフル活用）
- **GraphQLの学習**（Enum / Interface / Union / Custom Scalar / Connection / Dataloader / Directive まで網羅）
- **Railsポリモーフィック関連の学習**（通知機能で実装）
- **Rails各種パターンの学習**（STI / Concern / カスタムバリデータ / Scope / Counter Cache / enumrize / Delegated Types / has_many through）
- TDDで進める（RSpec + FactoryBot）

### コア思想

- カードを使う瞬間は**無思考**で動く（ショートカットが自動でPOSTする）
- 分類・確認はアプリを開いたときに**まとめてやる**
- 一般公開しない、自分だけが使う

## 📡 ショートカット → Rails POST仕様

### リクエスト

```json
POST /api/transactions
Header: X-API-Key: {固定キー}

{
  "transaction": {
    "amount": 1580,
    "store_name": "マクドナルド",
    "purchased_at": "2025-04-06T12:34:00+09:00"
  }
}
```

### Rails側の挙動

```
リクエスト受信
  → APIキー検証
    → store_nameでstore_category_mappingsを検索
      → 一致あり  : そのcategory_idで保存（source: :shortcut）
      → 一致なし  : category_id = null（未分類）で保存（source: :shortcut）
        → レスポンスを返す
```

### レスポンスとショートカット通知

|レスポンス                    |ショートカットが出す通知           |
|-------------------------|-----------------------|
|200 OK                   |「¥1,580 登録しました」        |
|401 Unauthorized         |「認証エラー：APIキーを確認してください」 |
|500 Internal Server Error|「登録失敗：あとで手動で入力してください」  |
|タイムアウト                   |「通信エラー：ネットワークを確認してください」|

- **リトライなし**（失敗したら通知だけ出して手動入力）
- 通知はRailsではなく**ショートカット側**が出す（APNs不要）

## 🗄 DBスキーマ

```sql
-- 支出レコード
transactions
  id
  amount          -- 金額（0以上の整数）
  store_name      -- 店名
  category_id     -- カテゴリ（null = 未分類）
  memo            -- メモ
  purchased_at    -- 購入日時
  source          -- 登録方法（shortcut / manual）※ enumrizeで管理
  created_at

-- カテゴリ（STI: VariableCategory / FixedCategory）

  type            -- STI用（"VariableCategory" / "FixedCategory"）
  name            -- カテゴリ名
  color           -- グラフ・UI表示用の色
  parent_id       -- 今はnull固定、将来の階層化用に仕込む
  sort_order      -- 並び順
  transactions_count -- Counter Cache（支出件数）

-- 店名→カテゴリの学習テーブル

  store_name      -- 店名
  category_id     -- カテゴリ

-- カテゴリ別・月別予算

  id
  category_id     -- カテゴリ
  amount          -- 予算額
  month           -- 対象月（例: 2025-04）

-- 予算アラート設定
alert_settings
  id
  category_id     -- null なら全体予算
  threshold       -- しきい値（例: 80 = 80%）※ユーザーが自由に設定
  is_active       -- ON/OFF

-- アプリ内通知（ポリモーフィック関連）
notifications
  id
  notifiable_type -- "BudgetAlert" / "UnclassifiedAlert"
  notifiable_id
  read_at         -- null = 未読、日時あり = 既読
  created_at

-- 予算アラートの実体（notifiable）
budget_alerts
  id
  category_id     -- 対象カテゴリ
  usage_percent   -- 発生時点の使用率
  threshold       -- 発生時点のしきい値
  created_at

-- 未分類アラートの実体（notifiable）
unclassified_alerts
  id
  count           -- 発生時点の未分類件数
  created_at
```

### カテゴリ設計の方針

- **STI（Single Table Inheritance）** で変動費・固定費を表現する
  - `VariableCategory`：食費・日用品など月によって変動する支出
  - `FixedCategory`：家賃・サブスクなど毎月ほぼ一定の支出
- 固定費カテゴリは**予算の扱い方が変わるだけ**（ペース判定の基準が変動費と異なる）
- **Phase 1から2階層構造で実装する**（`parent_id`を最初から活用）
  - 全カテゴリを2階層にできる（子カテゴリは任意で追加）
  - 初期は食費だけ子カテゴリあり（外食・自炊）
- **予算は親カテゴリにのみ設定する**
  - 子カテゴリの支出は親カテゴリの消費として集計
  - 子カテゴリはホーム画面で内訳として参考表示
- `transactions_count` で Counter Cache（高速な件数取得）

### 初期カテゴリ一覧

アプリ初回起動時にシードデータとして投入する。

| カテゴリ名 | 親カテゴリ | 種別 | 備考 |
|---------|-----------|------|------|
| 食費 | なし（親） | 変動費 | 予算設定対象 |
| 外食 | 食費 | 変動費 | 参考表示のみ |
| 自炊・スーパー | 食費 | 変動費 | 参考表示のみ |
| 日用品 | なし（親） | 変動費 | 予算設定対象 |
| 交通費 | なし（親） | 変動費 | 予算設定対象 |
| 娯楽 | なし（親） | 変動費 | 予算設定対象 |
| 家賃 | なし（親） | 固定費 | 予算設定対象 |
| サブスク | なし（親） | 固定費 | 予算設定対象 |
| その他 | なし（親） | 変動費 | 未分類の受け皿 |

### Union + ポリモーフィック関連（通知機能）

ホーム画面の通知エリアに使用する。GraphQLのUnion型とRailsのポリモーフィック関連を組み合わせて実装する。

**Railsモデル（ポリモーフィック）**

```ruby
class Notification < ApplicationRecord
  belongs_to :notifiable, polymorphic: true
  scope :unread, -> { where(read_at: nil) }
  scope :recent, -> { order(created_at: :desc).limit(20) }

  def read!
    update!(read_at: Time.current)
  end
end

class BudgetAlert < ApplicationRecord
  belongs_to :category
  has_one :notification, as: :notifiable
end

class UnclassifiedAlert < ApplicationRecord
  has_one :notification, as: :notifiable
end
```

**GraphQL（Union型）**

```graphql
# NotifiableContent は Union型
union NotifiableContent = BudgetAlert | UnclassifiedAlert

type Notification implements Node & Timestamped {
  id: ID!
  createdAt: ISO8601DateTime!
  readAt: ISO8601DateTime
  isRead: Boolean!
  notifiable: NotifiableContent!  # ポリモーフィックな実体
}

type BudgetAlert {
  category: Category!
  usagePercent: Float!
  threshold: Int!
}

type UnclassifiedAlert {
  count: Int!
}
```

**DataloaderでN+1対策**

```ruby
# app/graphql/sources/notifiable_source.rb
class Sources::NotifiableSource < GraphQL::Dataloader::Source
  def initialize(notifiable_type)
    @model = notifiable_type.constantize
  end

  def fetch(ids)
    records = @model.where(id: ids).index_by(&:id)
    ids.map { |id| records[id] }
  end
end

# NotificationType内
def notifiable
  dataloader.with(Sources::NotifiableSource, object.notifiable_type)
            .load(object.notifiable_id)
end
```

## ⚙️ Rails機能の使い方

### Active Storage

- 支出（transaction）に**写真を複数枚添付**できる
- 用途はレシートではなく**思い出・記録**（温泉の写真など）
- ブラウザからアプリ上で手動アップロード

```ruby
class Transaction < ApplicationRecord
  has_many_attached :photos
end
```

### Action Mailer + Active Job + Sidekiq

**月次サマリーメール（毎月1日に前月分を送信）**

含む内容：

1. 今月の総支出
1. カテゴリ別支出と予算比較
1. 前月との比較
1. よく行った店ランキング
1. 1日あたりの平均支出
1. **来月の推奨予算額**（直近3ヶ月の実績平均から自動計算）

**予算アラートメール**

- しきい値はユーザーが設定画面で自由に設定できる（デフォルト: 80%）
- カテゴリ別・複数設定可

### STI（Single Table Inheritance）

変動費・固定費でカテゴリを分類する。

```ruby
# app/models/category.rb
class Category < ApplicationRecord
  belongs_to :parent,   class_name: 'Category', optional: true
  has_many   :children, class_name: 'Category', foreign_key: :parent_id

  has_many :transactions
  has_many :budgets
  has_many :store_category_mappings
  has_many :stores, through: :store_category_mappings

  scope :root,     -> { where(parent_id: nil) }
  scope :children, -> { where.not(parent_id: nil) }

  def root?; parent_id.nil?; end
  def leaf?; children.none?; end

  # 子カテゴリを含む合計支出（予算進捗に使う）
  def total_spent_including_children(month)
    target_ids = [id] + children.pluck(:id)
    Transaction.by_month(month).where(category_id: target_ids).sum(:amount)
  end
end

# app/models/variable_category.rb
class VariableCategory < Category
  # ペース判定あり（GREEN/YELLOW/RED）
  # 月によって支出額が変動する
end

# app/models/fixed_category.rb
class FixedCategory < Category
  # ペース判定なし（固定額として扱う）
  # 毎月ほぼ一定の支出
end
```

### Concern（共通ロジックの切り出し）

```ruby
# app/models/concerns/summarizable.rb
module Summarizable
  extend ActiveSupport::Concern

  included do
    scope :this_month, -> { where(created_at: Time.current.beginning_of_month..) }
  end

  def monthly_total(month)
    transactions.where(purchased_at: month.all_month).sum(:amount)
  end
end

class Category < ApplicationRecord
  include Summarizable
end
```

```ruby
class FutureDateValidator < ActiveModel::EachValidator
  def validate_each(record, attribute, value)
    if value.present? && value > Time.current
      record.errors.add(attribute, "は未来の日時を指定できません")
    end
  end
end
```

### enumrize

Rails標準の`enum`（整数保存）ではなく、**文字列保存・i18n対応**のenumerizeを使う。

```ruby
class Transaction < ApplicationRecord
  extend Enumerize
  enumerize :source, in: [:shortcut, :manual], default: :shortcut
end

# 使い方
transaction.source          # => "shortcut"
transaction.source.shortcut? # => true
Transaction.with_source(:manual)  # scope自動生成

# GraphQL Enumとの役割の違い
# - enumrize: ActiveRecordレベルでのEnum管理（DB保存・スコープ・i18n）
# - GraphQL Enum: スキーマレベルでの型定義（クライアントとの契約）
```

### has_many through

```ruby
class Transaction < ApplicationRecord
  belongs_to :category
  has_one :budget, through: :category  # そのカテゴリの予算に直接アクセス
end

class Category < ApplicationRecord
  has_many :transactions
  has_many :budgets
  has_many :store_category_mappings
  has_many :stores, through: :store_category_mappings  # 登録済み店名一覧
end

# 使い方
transaction.budget  # カテゴリを経由して予算を取得（中間テーブル不要）
category.stores     # 店名一覧をワンステップで取得
```

### Delegated Types（Phase 3）

ポリモーフィック関連（Phase 2）で実装した通知を、Rails 6.1以降の Delegated Types にリファクタリングする比較学習。

```ruby
# ポリモーフィックとの違い
# ポリモーフィック: notifiable_type / notifiable_id の2カラム
# Delegated Types: notifiable_type / notifiable_id + 専用テーブルが必要

class Notification < ApplicationRecord
  delegated_type :notifiable, types: %w[BudgetAlert UnclassifiedAlert]
end

class BudgetAlert < ApplicationRecord
  include Notification::Notifiable
end
```

### ポリモーフィック関連（通知機能）

`Notification` モデルが `BudgetAlert` と `UnclassifiedAlert` の両方を `notifiable` として持つ。

```ruby
# 予算アラート発生時
alert = BudgetAlert.create!(category: category, usage_percent: 85, threshold: 80)
Notification.create!(notifiable: alert)

# 未分類アラート発生時
alert = UnclassifiedAlert.create!(count: 3)
Notification.create!(notifiable: alert)

# 未読通知の取得
Notification.unread.recent
# => [#<Notification notifiable_type="BudgetAlert" ...>, ...]
```

既読管理：
- `read_at: nil` = 未読
- `read_at: DateTime` = 既読

## 💰 予算機能の仕様

### 基本設計

- **カテゴリ別に予算を設定**する（月全体の予算は全カテゴリの合計として算出）
- 予算は**毎月自動引き継ぎ**（一度設定したら毎月同じ額が使われる）
- 変更時は「**今月のみ変更 / 以降すべて変更**」を選択できる
- 予算期間は**月初（1日）リセット固定**

### 予算設定UI

- **親カテゴリにのみ予算を設定できる**（子カテゴリには設定ボタンを表示しない）
- 予算ページでカテゴリのバーをタップ → 編集シートが出てくる
- 未設定のカテゴリには「＋ 予算を設定」と表示される

### 1日あたり使える額の計算ロジック

```
残り日数 = 月末日 - 今日 + 1  ※今日を含む
残り予算 = そのカテゴリの予算 - そのカテゴリの今月の支出
1日予算  = 残り予算 ÷ 残り日数
```

- **全体・カテゴリ別の両方**を表示する
- 予算オーバー（残り予算がマイナス）のときは **マイナス表示**する（例: -¥228/日）
- **週次ペース**（今週の支出 vs 週あたり理想額）も併せて表示する

### ペース判定ロジック

```
理想消費率 = 今日が月の何%経過しているか
実際消費率 = 今月の支出 ÷ 月予算

GREEN  : 実際消費率 < 理想消費率 × 0.9（余裕あり）
YELLOW : 理想消費率 × 0.9 ≦ 実際消費率 ≦ 理想消費率 × 1.1（ほぼ予定通り）
RED    : 実際消費率 > 理想消費率 × 1.1（使いすぎ）
```
