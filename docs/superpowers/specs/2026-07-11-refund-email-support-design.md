# 返金・取消メール対応 設計書

日付: 2026-07-11
ステータス: 承認済み（両対応）

## 背景（実メール調査に基づく）

三井住友カードの返金系通知は2形式ある（Gmail の実メールで確認済み）:

1. **取消（ご利用のお知らせ・既存対象件名）**: 承認取消の速報。
   `◇利用取引：取消` / `◇利用金額：-1,080円`（半角マイナス+カンマ+円）
2. **返品（ご利用明細のお知らせ・別件名）**: 明細確定時の差分通知。過去1年で2通のみ。
   `◇利用日：2025/10/03`（時刻なし）/ `◇利用取引：返品` / `◇利用金額：-264円`。
   定型文に「メール1通あたり30件」とあり **1通に複数明細**があり得る
   （実例: 返品-1,080 と 買物+1,080 の相殺ペア2行）。

**現行の実バグ**: 金額正規表現 `([\d,]+)\s*円` がマイナス記号を無視するため、
取消メールが来ると **+1,080円の支出として二重計上**される。

**重複リスクの検証結果**: 明細メールは速報の再送ではなく差分通知。取消済み取引は
「返品 + 買物」の相殺ペアで再掲されるため、速報・明細の両方を素直に全部取り込んでも
合計金額は一致する（観測された全ケースで確認）。速報済み買物の単純再掲は観測ゼロ。
残リスク（例: 海外利用のレート確定再掲）は運用で監視する（明細メールは年2通レベル）。

## 変更内容

### 1. パーサー（`src/lib/email-parser.ts`）

- **対象件名の拡張**: `ご利用のお知らせ` に加えて `ご利用明細のお知らせ` も対象にする
  （`subject.includes("ご利用のお知らせ")` は「ご利用明細のお知らせ」に一致しないため明示追加）
- **符号対応**: 円表記 `(-?)([\d,]+)\s*円`、JPY 表記も同様に `-?` を許可。
  マイナスなら amount を負数で返す
- **複数明細対応**: 新関数
  ```ts
  export function parseSmbcEmailItems(input: { from; subject; plain }):
    | { ok: true; items: { amount: number; storeName: string; purchasedAt: Date }[] }
    | { ok: false; reason: "not_target" | "parse_error"; error?: string }
  ```
  本文を `◇利用日` の出現位置でブロック分割し、各ブロックから
  利用日（時刻なし可）・利用先・利用金額を抽出する。
  - 1ブロックでも必須フィールドが欠けたら全体を parse_error（部分取り込みしない。
    失敗メールとして人間が確認する方が安全）
  - 既存 `parseSmbcEmail` は `parseSmbcEmailItems` の先頭1件を返す互換ラッパーに再構成
    （利用箇所が単一アイテム前提のため残す。`extractSmbcFields` は無変更）

### 2. Webhook（`src/app/api/inbound-email/route.ts`）

- `parseSmbcEmailItems` を使い、**全アイテムを単一DBトランザクションで**取引化:
  - 各アイテム: 店名マッピング照合 → 取引 insert → `evaluateAlertsForTransaction`
  - 最後に `refreshUnclassifiedAlert` を1回
  - `inbound_emails.transactionId` は**先頭の取引**に紐付け（現行スキーマは1対1のため。
    複数明細メールは年2通レベルなので join テーブル追加はしない = YAGNI）

### 3. 影響確認（変更不要と判断した箇所）

- 集計（monthly-summary / alerts / budget-pace）: 全て sum ベースで負額が自然に相殺される
- 表示: `¥{amount.toLocaleString()}` は負数を `¥-1,080` と表示（許容）
- 手動取引フォーム・失敗メール解決: 正数バリデーションのまま（返金の手入力は今回対象外）
